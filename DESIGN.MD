# TinyCIC Design

## 开发原则

- **单一状态管理** - DESIGN.MD 是项目唯一的状态管理器，所有状态变更必须在此记录
- **功能独立提交** - 每个功能必须单独提交，确保变更可追溯
- **测试驱动** - 不写传统测试，每次提交前确认所有定理通过类型检查和对应的破坏检查
- **代码不写注释** - 用清晰的命名和逻辑代替注释

## 破坏检查

主动破坏检查是一种验证类型检查器正确性的测试方法：故意编写语义错误的证明，验证系统能否正确识别并拒绝。

### 实施步骤

1. **选择中等复杂度的通过类型检查的证明**
   - 包含多个参数（如 `a : Rat`, `ha : gt a zero`）
   - 目标是命题（`Prop`）而非类型

2. **故意构造类型错误**
   - 将 `exact ha` 改为 `exact a`
   - `a` 的类型是 `Rat`，目标的类型是 `Prop`
   - 这是明显的类型不匹配

3. **类型检查**
   - 结果应该是类型检查失败

## 开发目标

- 复用 Lean 4的cpp的核心系统(inductive, environment, type checker)
- 可信内核尽量小
- 更适合 AI 生成显式 proof term
- 保留人类可读的推理结构
- 设计X语言，文件后缀x，语法是简化版本的lean风格，不用实现复杂的宏和tactic功能，简单实现import/have/inductive/theorem/def等关键字， 不要实现sorry这个语义的关键字
- 完成cauchy序列来构造实数(大概步骤: 逻辑基础，整数，有理数，柯西序列，实数)，完成完备性证明(逻辑基础与大学数学分析课程保持一致)

## 开发流程

1. 将Lean4作为git submodule，不修改源码
2. 将lean4 中(kernel + runtime + util)编译成静态库libkernel.a
3. libkernel.a中保留了写stub，这些stub按需用rust实现缺失符号
4. 然后完成x的实现，采用rust
